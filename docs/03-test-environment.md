## 3.2 결제 Mock 서버 스펙

| 항목 | 내용 |
|---|---|
| 응답 지연 | 1,000ms ~ 2,000ms (균등 분포 랜덤) |
| 성공률 | 약 95% (5% 확률로 실패 응답 반환) |
| 실패 응답 | HTTP 200 + `{"status": "FAIL"}` 또는 HTTP 500 |
| 역할 | 실제 외부 PG사의 네트워크 I/O 및 불확실성을 재현 |

## 3.3 부하 테스트 스크립트 시나리오

**시나리오 A — 정합성 검증 (Correctness Test)**

| 항목 | 값 |
|---|---|
| 가상 사용자(VU) | 5,000 |
| 요청 패턴 | 동시에 예매 API 1회씩 호출 |
| 목적 | 티켓 초과 판매 여부, DB 정합성 확인 |

**시나리오 B — 지속 부하 (Sustained Load Test)**

| 항목 | 값 |
|---|---|
| 가상 사용자(VU) | 500 |
| 지속 시간 | 60초 |
| 요청 패턴 | 초당 반복 예매 시도 |
| 목적 | 지속 트래픽 하 TPS 및 응답 시간 측정 |

**시나리오 C — 램프업 (Ramp-up Test)**

| 항목 | 값 |
|---|---|
| 가상 사용자 | 0 → 1,000 → 5,000 (단계적 증가) |
| 목적 | 병목 지점 탐지, 서버 한계 TPS 파악 |

## 3.4 성공 기준 (Pass/Fail Criteria)

| 검증 항목 | 기준 | 측정 방법 |
|---|---|---|
| 정합성 — 예매 건수 | DB `reservation` 테이블 레코드 정확히 1,000건 | SQL `COUNT(*)` |
| 정합성 — 잔여 수량 | `concert.remaining_tickets = 0` | SQL 조회 |
| 초과 판매 없음 | 1,001건 이상 저장 시 실험 실패 | SQL `COUNT(*)` |
| 가용성 — 응답 수신율 | 5,000건 중 응답(성공 or 실패) 수신 비율 ≥ 80% | 부하 도구 리포트 |
| 응답 시간 P99 | 가능한 낮게 (단계별 최적화로 기록) | 부하 도구 리포트 |
| 서버 안정성 | 테스트 중 OOM / 프로세스 종료 없음 | 서버 모니터링 |
