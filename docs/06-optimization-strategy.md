# 버전별 최적화 전략

각 버전은 동일한 비즈니스 조건에서 서로 다른 레이어의 최적화 전략을 적용하여,
처리량과 병목의 변화를 비교합니다.

## v0. Baseline (실패 케이스)

| 구분 | 내용                                 |
|---|------------------------------------|
| 코드 | DB 락을 통한 최소한의 동시성 제어 + naive 구현    |
| 아키텍처 | 단일 애플리케이션 + DB                     |
| 인프라 | 단일 저사양 서버                          |
| 특징 | Oversell 발생 또는 TPS 목표 미달, 병목 구조 확인 |

---

## v1. 소프트웨어 최적화 (단일 서버)

| 구분 | 내용 |
|---|---|
| 코드 | 트랜잭션 범위 조정,  불필요한 쿼리 제거 |
| 아키텍처 | 변경 없음 (단일 애플리케이션 구조 유지) |
| 인프라 | 단일 저사양 서버 유지 |
| 특징 | 인프라 확장 없이 처리량 개선, lock contention 및 thread/connection 병목 관찰 |

---

## v2. 아키텍처 최적화

| 구분 | 내용 |
|---|---|
| 코드 | 비동기 처리 도입, 요청 흐름 분리 |
| 아키텍처 | Redis 기반 선점 처리, Queue를 통한 비동기 처리 구조 도입 |
| 인프라 | 단일 서버 (추가 컨테이너: Redis 등) |
| 특징 | DB 병목을 완화하고 I/O 대기 시간 분산, 구조적 병목 이동 관찰 |

---

## v3. 인프라 확장 (확장 실험)

| 구분 | 내용 |
|---|---|
| 코드 | 상태 공유를 고려한 로직 (stateless 설계) |
| 아키텍처 | 다중 인스턴스 + 로드밸런싱 |
| 인프라 | NCP 기반 다중 노드 구성 |
| 특징 | 수평 확장을 통한 처리량 증가, 대신 정합성 관리 복잡도 증가 및 비용 대비 효율 분석 |
