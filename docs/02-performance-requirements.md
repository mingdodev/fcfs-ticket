# 성능 요구사항 (The Challenge)

## 2.1 환경 제약

| 항목 | 내용 |
|---|---|
| 서버 구성 | 단일 NCP 인스턴스 (최저 사양) |
| 컨테이너 | 서비스 서버 인스턴스에서 Docker로 Spring Boot 앱 + MySQL 컨테이너 각각 기동 |
| 네트워크 | 부하 서버 ↔ 서비스 서버: 같은 VPC 내부망 |
| 서비스 서버 ↔ 결제 Mock 서버: 같은 VPC 내부망 |

## 2.2 트래픽 목표 (동시성, TPS)

| 항목 | 값 |
|---|---|
| 동시 접속 사용자 | 5,000명 |
| 총 예매 시도 건수 | 5,000건 |
| 오픈 직후 폭증 패턴 | 수 초 내 전량 요청 집중 |
| 목표 처리 TPS | 100~200 TPS |

## 2.3 절대 원칙 (정합성)

> **"서버는 죽어도 데이터는 죽지 않는다"**

- 티켓이 1,001장 이상 팔리는 **초과 판매(Oversell)는 절대 허용하지 않는다.**
- 결제 실패 또는 응답 없음 시 예매 내역이 DB에 남아서는 안 된다.
- 동시 요청 간 Race Condition으로 인한 정합성 오류가 없어야 한다.

## 2.4 가용성 목표 (오류율)

> **"최대한 많이 수용한다"**

- 5,000건의 요청 중 타임아웃으로 튕겨 나가는 비율을 최소화한다.
- 처리 불가 요청에는 명확한 실패 응답(예매 마감, 결제 실패 등)을 반환한다.
- 서버 자원(스레드, 커넥션 풀)을 최대한 활용해 대기 시간을 줄인다.

<br>

**정합성을 절대 원칙으로 두고, 동시성과 가용성 사이 어떤 트레이드 오프가 있는지 관찰한다.**
