# 비즈니스 상황 (The Scenario)

## 1.1 서비스 개요

한정 수량(1,000장) 티켓을 선착순으로 판매하는 공연 예매 서비스.
오픈 직후 수천 명의 사용자가 동시에 예매를 시도하는 고트래픽 상황을 가정한다.

## 1.2 핵심 흐름

```
사용자 요청
    │
    ▼
[1] 잔여 티켓 확인          ← 정합성 핵심 지점 (Race Condition 발생 가능)
    │
    ├─ 티켓 없음 → 예매 실패 (sold-out) 응답
    │
    └─ 티켓 있음
         │
         ▼
[2] 외부 결제 API 호출       ← I/O 병목 지점 (1~2초 Delay)
         │
         ├─ 실패/타임아웃 → 재시도 후 예매 취소 (결제 없음)
         │
         └─ 성공
              │
              ▼
         [3] 예매 내역 저장 + 티켓 수량 -1   ← 원자적 처리 필요
              │
              ▼
         예매 완료 응답
```

## 1.3 도메인 모델 (간략)

| 엔티티 | 주요 속성 |
|---|---|
| `Concert` | id, name, remainingTickets |
| `Reservation` | id, concertId, userId, createdAt |

> 무통장 입금과 같이 대기가 필요한 결제 수단은 제공하지 않는다고 가정한다. 또한, 좌석 선택과 같은 세부 과정은 구현하지 않고, 정합성/가용성/동시성 사이 트레이드오프를 관찰하기 위한 최소 기능에 집중한다.
> `Reservation` 레코드의 존재 자체가 예매 성공을 의미한다. 결제 실패 시 레코드를 저장하지 않으므로 별도 status 필드는 불필요하다.
